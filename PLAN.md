# atlasent-sdk — Coordinated Release Plan

## Role
The SDK is the ONLY component that gates execution. It wraps protected actions in an execution wrapper that calls /v1/evaluate and /v1/verify-permit before allowing any protected action to proceed.

## Core Enforcement Contract (Non-Negotiable)
The execution wrapper implements this decision table — no exceptions:

| verify-permit response | SDK action |
|---|---|
| HTTP 200, status = "valid" | invoke protected action (ONLY valid path) |
| HTTP 200, status ≠ "valid" | BLOCK |
| HTTP 200, status unrecognized | BLOCK |
| HTTP 4xx (any) | BLOCK |
| HTTP 5xx (any) | BLOCK |
| Network timeout | BLOCK |
| Network error | BLOCK |
| Response unparseable | BLOCK |
| No response | BLOCK |

## v1 Release State (v1.5.0)
- evaluate() — calls POST /v1/evaluate
- authorize() / withPermit() — execution wrapper (calls verify-permit before action)
- Single-use permit enforcement
- Retry with idempotency_key preservation (ARCH-10)
- Clock skew tolerance ±5s (ARCH-09)
- JWKS caching ≤60s with refresh on unknown kid (ARCH-12)
- context_hash canonical serialization (ARCH-04)
- action_instance_id UUID v4 generation (ARCH-05)

## SDLC Constraints Applied
- ARCH-01: Protected actions only callable through execution wrapper
- ARCH-05: action_instance_id generated by SDK (UUID v4); never caller-supplied without validation
- ARCH-09: Clock skew ±5s on exp validation
- ARCH-10: Retry preserves original idempotency_key and action_instance_id
- ARCH-12: JWKS refresh on unknown kid
- NFR-01/02/03: Fail-closed on all error conditions
